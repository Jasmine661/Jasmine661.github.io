重绘与回流

发表于2025-09-23|更新于2025-09-25|前端性能优化重绘与回流
|浏览量:

概念

引入

浏览器的主要线程有 JS线程 和 UI线程，宏观顺序关系是 JS 执行 → 样式计算 → 回流 → 重绘 → 合成 →GPU→ 下一轮循环，所以如果一个循环中进行多次的回流重绘，就会阻塞下一个循环的 JS执行

回流（Reflow / Layout）

当浏览器发现 render tree 中某个部分的几何尺寸（位置、大小）发生变化，需要重新计算元素的几何属性，然后重新布局整个渲染树，这个过程叫做回流。

触发回流的典型操作：

  • 添加或删除可见的 DOM 元素
  • 元素的大小、位置发生变化（比如 width、height、margin、padding、border、font-size 等属性改变）
  • 浏览器窗口大小变化（因为会影响布局）
  • 获取某些属性时（如 offsetTop、offsetHeight、clientWidth、scrollTop 等），因为需要计算最新的布局结果

回流的代价比较大，因为可能会导致整个页面的重新布局。

重绘（Repaint）

定义：当元素的外观样式改变，但没有影响布局时，浏览器会重新把该元素绘制一遍，这个过程叫做重绘。

触发重绘的典型操作：

  • 改变元素的 color、background-color、visibility、box-shadow 等不会影响几何尺寸的属性

重绘比回流开销小，但仍然会消耗性能。

减少重绘回流的方案

1.一般性的优化思路

(1) 减少回流：

避免逐条修改样式

批量 DOM 操作：先 documentFragment 中处理，最后一次性塞进 DOM

缓存需要的布局信息：避免在循环里多次读取 offsetHeight / clientWidth，因为每次读取都会触发回流

(2) 减少重绘

只修改必要的样式，比如动画尽量用 transform 和 opacity（它们不会触发回流，通常只会触发合成）

2.利用“图层隔离”（GPU 合成层）

Chrome创建图层的条件

1.3D 或透视变换

2.使用will-change提示浏览器

3.使用CSS动画或过渡且涉及合成属性

​ 当元素正在进行 transform / opacity 动画时，浏览器会自动把它提升为独立图层，避免主线程频繁计算布局

4.拥有 CSS 滤镜或遮罩

  • 使用 filter（模糊、亮度、对比度等）
  • 使用 mask 或 clip-path

5.使用 video、canvas、iframe 等媒体元素

6.具有特定的 CSS 属性

  • position: fixed（在一些场景下可能被单独提取出来，方便滚动合成）
  • overflow: scroll 或 overflow: auto 且内容溢出时（独立的滚动层）

7.被 GPU 加速绘制要求覆盖

实战优化

用 translate 代替 top 改变

top 会触发 Layout 的过程但是 translate 不会。但是现在浏览器对position: absolute/fixed + top/left的优化已经好了很多，在简单动画中，top的消耗也是比较小的。但是最推荐的还是使用translate。

用 opacity 代替 visibility

opacity 和 visibility都不会触发回流。注意有opacity属性的元素在独立图层的时候，修改opacity只会触发合成，不会触发重绘，但如果元素在普通的图层，仍会触发重绘。但是我在这里遇到一些问题，在 performance 中看还是会有显示了重绘**（有待解决）**，但是我测试 opacity 时没有观察到闪绿，在测试 visibility 时还是会闪绿的。

而且几次测试下开，感觉两个的性能差别不大？很可能是我的demo比较简单的原因，在读网上的文章中还是更推荐使用 opacity。

不仅仅有上面两个例子可以在重绘回流方面进行优化，还有很多CSS属性可以做到减少重回回流这个需求**（待探索）**

批量修改DOM的class

一条一条的修改 DOM 样式，每次修改都会进行一次重绘，定义在一个className中进行一次性修改，用add、append、remove等。有的时候一条一条的修改class并没有出现修改一次只从一次回流重绘，是因为浏览器自带的缓冲机制

避免将DOM属性值在循环中作为变量

获取 DOM 节点的几何属性值，如 offsetHeight、offsetWidth，时会触发一个回流的过程。因为浏览器回去获取当前最新的某个 DOM 元素在我整个布局中的位置，就会强制刷新缓冲区域。缓冲机制值为了避免回流过于频繁的，将 DOM 节点的属性值放在循环里就会不断地破坏缓冲机制，使其失效

避免使用 table 布局

​ table 中很小的改动会影响整个 table ，当 table 很大的时候会增加 Layout 的时间，所以尽量去使用 div 布局，因为某个 div 需要重新布局的话，只会影响后面的元素的布局

动画实现的速度选择

​ 动画触发的代价取决于动画属性：layout > paint > composite。作用在layout属性会回流，作用在paint属性会重绘，作用在composite属性不会回流重绘，只会在合成阶段运行。如果动画的频率过高就会一直回流重绘重新获取位置信息等，就会导致浏览器很卡顿，即 JS 线程的执行会受到 UI 线程的阻塞，而且重绘和计算也是会消耗CPU资源的。

对于动画新建图层

​ 对于video、canvas、transform、will-change这样属性的 DOM 元素，chrome会自动把它设置为一个新的图层。但是对于自己写的动画可能使用 div 包裹的，chrome不知道是动画，就不会创造新的图层，所以需要手动创建。

启用 GPU 硬件加速

​ 在我们使用位置变换的时候，如果用 translate 3D，会涉及到 GPU 的运算能力。GPU 的运算能力对于整个纹理的绘制，整个图像的并行运算其实是非常有好处的。因为 GPU 本质是一个并行运算的运算单元，对需要进行并行运算的图像的处理速度是非常快的。但是 GPU 加速也是有代价的，使用 GPU 加速意味着数据要从 CPU 走总线到 GPU 上， GPU 才能对数据进行运算，代价主要体现在数据从 CPU 走到 GPU 的损耗。

如何启动 GPU加速？

  • 使用 3D 变换强制开启 GPU

    transform: translateZ(0); transform: translate3d(0,0,0);

  • 使用合成级别的动画属性

    transform opacity

  • 使用 will-change 提示

    注意不要滥用will-change，Chrome对其的滥用会有警告

  • 使用 CSS 滤镜 / 特效

    filter: blur(10px); backdrop-filter: blur(5px); mix-blend-mode 这些计算开销大，浏览器会放在 GPU 加速

  • 使用视频 / canvas / iframe / WebGL

    默认就会创建独立合成图层，走 GPU

写在最后

任何一个优化都会有相应的代价，我们在做性能优化的时候应该根据业务系统去balance
