JPG图片的解析过程——有损压缩

颜色空间转化 -> 分块处理 -> 离散余弦变换(DCT) -> 量化(Quantization，有损压缩关键) -> 熵编码(无损)

png8、png24、png32之间的区别

png格式本身是一种颜色索引，.png的图片文件除了颜色块本身，还有一些png的编码

  • png8 —— 236色，支持透明
  • png24 —— 2^24色，不支持透明
  • png32 —— 2^24色，支持透明

每种png格式的文件都有其自己的特点，我们可以根据自己对文件大小和颜色丰富程度等因素来选择使用哪一种png格式

不同格式图片常用的业务场景

  • jpg/jpeg 有损压缩，压缩率高，不支持透明 —— 大部分不需要透明图片的业务场景
  • png 无损压缩，支持透明，浏览器兼容性好，但是大小通常比jpg大 —— 大部分需要透明图片的业务场景
  • gif 支持256色，动图和简单的透明(全透明或者不透明) —— 表情包、动图、小型广告
  • webp 压缩程度更好，但在ios14以下有兼容性问题 —— 安卓全部
  • svg 矢量图，代码内嵌，相对较小，图片样式相对简单的场景 —— 图片样式相对简单的业务场景

图片压缩的方法

CSS雪碧图(过时)

通过将网站上要用到的图片整合到一张单独的图片上

优点：可以减少 HTTP 请求数量

缺点：如果图片比较大，一次加载会比较慢

Image inline

将图片的内容转化为 base64 的格式内嵌到 html 中

优点：减少 HTTP 请求数量

使用矢量图

使用 SVG 进行矢量图绘制，使用 iconfont 解决icon的问题

使用webp

WebP 的优势体现在它具有更优的图像数据压缩算法能带来更小的图片体积，而且拥有肉眼识别无差异的图像质量;同时具备了无损和有损的压缩模式、Alpha 透明以及动画的特性，在 JPEG 和 PNG 上的转化效果都非常优秀、稳定和统一。

构建工具中的压缩方式

Webpack

使用webpack-image-loader

Vite

使用vite-plugin-imagemin

以上两个构建工具中对 png 都有 optipng 和 pngquant ，前者是做无损压缩，优化 PNG 的编码方式，后者是有损压缩。两者通常一起配合使用。

Rollup

使用 rollup-plugin-imagemin

单独压缩工具

在CI/CD阶段处理，而不是在前端构建时压缩（推荐在CI/CD时统一压缩，每次构建都压缩的话，构建速度会很慢）

  • imagemin（Node脚本批量压缩）
  • Squoosh CLI（Google出品）
