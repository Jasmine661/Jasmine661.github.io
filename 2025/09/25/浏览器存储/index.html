浏览器的存储方式介绍

cookie

因为 http请求 本身是无状态的，即请求结束之后连接就关闭了，下一次请求时服务端是不清楚这次请求和上次请求是不是来自同一个客户端的，即不知道两次请求有什么关系，就需要去解决关于用户信息的问题。

cookie就是为了用来维护客户端和浏览器之间的状态的标识，告诉服务端请求来自哪个客户端。是由服务端生成，交给客户端存储和管理，在每次请求的时候把cookie放在http header，服务端就会知道当前请求来自哪个客户端了。

我们也能通过 JS 的 document.cookie 去查看或者修改 cookie。但是如果 cookie 设置了 httponly，当前 cookie 就不支持 JS 读写

总的来说cookie是用于 浏览器端和服务器端的交互 以及 客户端自身数据的存储。

cookie的缺陷

1.作为浏览器存储，大小只有4kb左右

2.需要设置过期时间 expire

所以仅仅是做浏览器存储的话，用 localStorage 会更好

性能消耗

由于 cookie 是一个域名维度下的概念，所以只要是这个域名下的所有请求，都会携带 cookie，就容易造成 cdn 的流量损耗。所以我们要把 cdn 的域名和主站的域名独立开，在 cdn 上存储静态文件而不要携带 cookie。

LocalStorage、SessionStorage

LocalStorage 是 HTML5 设计出来专门用于浏览器存储的，大小为 5M 左右，仅在客户端使用，不和服务端进行通信。JS 读取使用 localStorage 的 API 有很好的封装。

性能优化点主要在浏览器的本地缓存方案，常用于减少首屏加载时间等等

SessionStorage 是 HTML5 设计出来的会话级别的浏览器存储，除此之外几乎和 LocalStorage 差不多。

性能优化点可以在表单信息的维护上，用于提升用户的体验

IndexedDB

是一种低级 API ，用于客户端存储大量结构化数据。该 API 使用索引来实现对该数据的高性能搜索。虽然 Web Storage 对于存储较少量的数据很有用，但对于存储更大量的结构化数据来说，这种方法不太有用。这个时候就可以使用 IndexedDB 来提供解决方案。

IndexedDB实际上在网络中的各大站点上的应用是比较少的，但是也确实可以做很多事。比如可以为我们的应用创建离线版本

PWA

PWA (Progressive Web Apps) 是一种Web App 新模型，并不是具体指某一种前沿的技术或者某一个单一的知识点。这是一个渐进式的 Web App，是通过一系列新的 Web 特性，配合优秀的 UI 交互设计，逐步的增强 Web App 的用户体验。

  • 可靠性：Web App在没用有网络的环境中也能够提供基本的页面访问，而不会出现“未连接互联网”的页面。这其实是 Web 相对 IOS 和 安卓 的一个劣势。Web App在前端在这一方面是增量式的从网络上加载资源的一个过程，如果没有网络就加载不到资源。
  • 快速：针对网页渲染以及网络数据访问有较好的优化，相较于原生App。
  • 融入(Engaging)：应用可以被增加到手机桌面，并且和普通应用一样有全屏、推进等特性。这一点是完全对标App的。

Service Workers

SW 是一个脚本，浏览器独立于当前网页，将其在后台运行 ，去实现一些不依赖页面或者用户交互的特性。

JavaScript 本身是一个单线程的，所有的 JS 代码都是在 JS 主线程去执行。当要进行非常大数据的读取，或者巨量的前端相关的运算的话，实际上会严重阻碍相关页面的渲染。

Service Workers 的作用就是对这样大规模数据在其他的线程去进行运算和获取，然后将结果返回到主线程，由主线程进行渲染或者执行其他后续的操作，这样可以避免主线程被 JS 代码巨大的逻辑阻塞，大大提升了浏览器在处理大规模 JS 运算的能力。

特性

  • 拦截和代理网络请求（Fetch Interception）

    可以拦截页面发出的 fetch / XHR 请求

    按策略返回缓存、网络资源，或者做离线回退

    这就是 离线体验 和 资源加速 的关键

  • 离线缓存

    搭配 Cache Storage API 使用，SW 可以把静态文件（HTML/CSS/JS/图片）缓存到本地

    即便断网，也能直接从缓存中取出资源。

  • 后台同步

    当用户离线时，操作会暂存在本地

    等网络恢复时再统一发送请求（比如离线发送消息）。

  • 推送通知

    SW 可以和服务器保持连接，接收 Web Push Notification

    即使页面关闭，也能在系统通知栏显示。

  • 定期后台同步

    可以让 SW 定时拉取内容（比如新闻更新、天气数据）。

    相比普通 Background Sync 更自动化。

  • 后台任务处理

    可以在用户没开网页时执行一些逻辑（比如静默更新缓存）。

生命周期

Active常见事件

  • Fetch：拦截网络请求（离线缓存策略）
  • push：接收推送
  • sync：后台同步

Chrome 中对 SW 的检查

以下是两个chrome的命令

  • chrome://serviceworker-internals/?devtools ：可以查看自己已经在 Chrome 中注册过的 SW 的情况，我这里常见的网站我是打开了百度云（https://cloud.baidu.com/）查看的

  • chrome://inspect/#service-workers ：可以查看现在浏览器正在开启 SW 的网站有哪些
